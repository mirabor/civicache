<threat_model>
## Trust Boundaries

Phase 6 Plan 07 introduces one new data surface: `docs/demo-backup.mov`
(a binary video file recorded on the user's laptop). Minimal security
implication — it is visual content of a terminal session, not machine input
for any downstream tool. `demo-rehearsal.log` is plain-text stdout + timestamps.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-07-01 | Information Disclosure | Screen recording may capture secrets visible in terminal (e.g. API keys printed by accident) | mitigate | Task 2 Step 7 explicitly asks user to verify playability AND that the demo.sh output is what appears — demo.sh does NOT echo env vars (confirmed by Plan 02 grep test); the only on-screen text is the 3 step banners + miss-ratio table + wall-clock. If user's shell prompt exposes secrets (rare), they should use a clean terminal session for recording |
| T-06-07-02 | Repudiation | demo-rehearsal.log could be fabricated to show 3 passes when only 1 was run | accept | Trust model is the honesty of the project author; no adversarial setting. The log shape (date + wall-clock + stdout) is hard to fake-fabricate convincingly; and the rehearsal artifact exists to serve the project author's own demo-day reliability |
| T-06-07-03 | Denial of Service | Screen recording file >10MB bloats repo if committed | mitigate | Task 2 Step 8 explicitly branches on 10MB threshold per PROCESS.md convention; Task 3 enforces the gitignore decision |
</threat_model>
