<threat_model>
## Trust Boundaries

This plan introduces a new trust boundary: the user's local machine now holds a CourtListener API token. The token crosses the boundary in one direction only — outbound from .env to the CourtListener API server via HTTPS.

| Boundary | Description |
|----------|-------------|
| .env file | Secrets-at-rest on local filesystem; gitignored; never committed |
| HTTPS to courtlistener.com | Authenticated token leaves machine over TLS in the Authorization header |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-13 | Information Disclosure | COURTLISTENER_API_KEY accidentally committed | mitigate | .gitignore line 45 excludes .env. .env.example has empty values only. Acceptance criteria explicitly check .env.example has no real values. |
| T-01-14 | Information Disclosure | Token echoed in logs / terminal history | mitigate | Task 3 extracts token via shell command substitution and redirects curl bodies to /tmp; never echoes the token itself. SUMMARY records HTTP codes only. |
| T-01-15 | Spoofing | Typo'd base URL (pacer.uscourts.gov etc) | mitigate | Plan hardcodes https://www.courtlistener.com for all curl tests; PITFALLS C1 flagged as top risk. |
| T-01-16 | Tampering | Token replaced by attacker with local write access | accept | Beyond Phase 1 threat model; local machine compromise is out of scope for a student project. |
| T-01-17 | Denial of Service | 5k/hr throttle tripped by verification | mitigate | Only ~5 curl calls with 1s sleep between; well inside the 5000/hr budget. |
</threat_model>
